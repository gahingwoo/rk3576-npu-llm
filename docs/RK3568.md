# Kiln on RK3568 (Radxa ROCK 3B) — vision NPU

**Status: implemented, NOT yet tested on real hardware.** The RK3568 path is
built the same way as the working RK3576 one (mainline kernel + Kiln
patches + the vendor `rknpu` module), but it has not run on a board yet. A
ROCK 3B is with **@midg971** for validation; until then treat everything here
as "should work", not "does work". The RK3576 path (ROCK 4D) is the tested one —
see [`MAINLINE-KERNEL.md`](MAINLINE-KERNEL.md).

> **Want to help test this?** If you have a ROCK 3B, running the installer and then
> `kiln-doctor` + `kiln-vision /opt/models/test.jpg` and reporting back in a
> [GitHub issue](https://github.com/gahingwoo/kiln/issues) (paste the `kiln-doctor`
> output) is the single most useful contribution right now — it's what turns "should
> work" into "does work". Wins, failures, and dmesg all welcome.

## Why a custom kernel (and not the simpler overlay route)

The out-of-tree [`w568w/rknpu-module`](https://github.com/w568w/rknpu-module)
runs the same vendor NPU driver on RK3566 with a **DT overlay on a stock
kernel**, but has to **disable the IOMMU** and run in non-IOMMU mode. The reason
is a real kernel bug: the RK3566/RK3568 IOMMU **v2** hardware cannot fetch a
page table located above 4 GB, and mainline `iommu_data_ops_v2` leaves
`.gfp_flags = 0`, so on a >4 GB board the NPU MMU bus-errors. That cannot be
fixed from a module.

Kiln builds a kernel, so it fixes it properly instead of disabling the MMU:
`kernel-patches-rk3568/0003` sets `GFP_DMA32` on the v2 ops (like v1), keeping
NPU page tables under 4 GB. The IOMMU is then **enabled**. This is the same
"custom kernel, control the variables, no mmu fail" approach used for RK3576.

## What's different from RK3576

| | RK3576 (ROCK 4D, tested) | RK3568 (ROCK 3B, untested) |
|---|---|---|
| NPU | dual core @ `27700000` | **single core @ `fde40000`** |
| `compatible` | `rockchip,rk3576-rknpu` | `rockchip,rk3568-rknpu` |
| Power domain | NPUTOP → NPU0/NPU1, needs settle-delay / mem-reset / BIU patches | **plain `RK3568_PD_NPU`, no special pmdomain patches** |
| NPU clock | CRU `CLK_RKNN_DSU0` @ 594 MHz | CRU `CLK_NPU` @ 800 MHz |
| `state_init` | `rk3576_state_init` (executer arming) | **NULL** (RK3568 needs none) |
| IOMMU | v2, works out of the box | v2 + **`GFP_DMA32` fix** (patch 0003) |
| Extra | `vdd_npu` fine | **`vdd_npu regulator-always-on`** or PD enable `-ETIMEDOUT`s |
| Workload | LLM + vision | **vision (RKNN)** — RK3568 is ~0.8 TOPS, LLM is impractical |
| Radio | onboard aic8800 (Kiln patches it) | different radio; installer skips aic8800 |

The vendor `rknpu` module is **the same binary** for both — it already carries
the `rockchip,rk3568-rknpu` match and RK3568 config (the `rk3568` `of_device_id`
in the fetched `rknpu_drv.c`; the port is described in
[`driver/patches/README.md`](../driver/patches/README.md)).

## Patches (`kernel-patches-rk3568/`)

| file | what |
|---|---|
| `0001-arm64-dts-rk3568-add-rknpu-node-and-npu-power-domain.patch` | adds `npu@fde40000` + IOMMU `@fde4b000` + `RK3568_PD_NPU` to `rk3568.dtsi` (disabled) |
| `0002-arm64-dts-rk3568-rock-3b-enable-npu.patch` | enables both on ROCK 3B + `vdd_npu regulator-always-on` |
| `0003-iommu-rockchip-force-gfp-dma32-for-v2-page-tables.patch` | the >4 GB IOMMU v2 fix |

Verified so far: all three apply to mainline 7.1.3 and `rk3568-rock-3b.dtb`
compiles with the NPU + IOMMU enabled. **Not** verified: it powers on, probes,
or runs a conv on real hardware.

## Install (once the kernel is built)

`kiln-install.sh` auto-detects the SoC from `/proc/device-tree/compatible` and
pulls the right kernel release (`kiln-mainline-kernel-rk3568`) + installs the
`rk3568-rock-3b.dtb`. On a ROCK 3B:

```sh
curl -fsSL https://raw.githubusercontent.com/gahingwoo/kiln/main/scripts/kiln-install.sh | bash
```

Same hands-off flow as the ROCK 4D: the script self-sudos, then **reboots itself
twice (~10–15 min)** into a ready system — don't cut power — and reports the result
at login. Verify with `kiln-doctor`. `KILN_MANUAL=1 bash …` keeps the reboots
hands-on. See [`ARMBIAN.md`](ARMBIAN.md).

Then vision (you supply the model — nothing is baked in):

```sh
# put a MobileNet .rknn built for target_platform="rk3568" in /opt/models
kiln-vision /opt/models/test.jpg
```

## Serve & configure (vision-only)

RK3568 is a **vision** target (~0.8 TOPS; LLM is impractical), so the productized
tools run in vision-only mode — no `.rkllm` needed:

- **`kiln-serve`** starts vision-only when there is no LLM model: it serves
  `POST /v1/vision/classify` (image → top-N) and answers `503` on
  `/v1/chat/completions`. Same binary as on RK3576.

  ```sh
  kiln-serve                              # "vision-only mode" if no .rkllm present
  curl http://<rock3b>:8080/v1/vision/classify?top_n=5 --data-binary @cat.jpg
  ```
- Configure the `[vision]` section (model / labels / top-N / `core_mask` /
  priority) and `[server]` by editing `/etc/kiln/config.ini`; the `[llm]` fields
  are simply unused here. See [`docs/CONFIG.md`](CONFIG.md) and
  [`docs/SERVER.md`](SERVER.md).

The installer (SoC-detected) seeds `/etc/kiln/config.ini` with the RK3568 vision
model; the LLM tools (`kiln-chat`) just have nothing to load on this board.

## Open items (for the ROCK 3B bring-up)

1. **On-board validation** — does the NPU power on (watch for `-ETIMEDOUT` /
   `failed to get pm runtime`), probe (`renderD*`), and run a conv with
   **correct** output (not uniform zero-point)?
2. **MMU fault IRQ** — the `iommu@fde4b000` node currently reuses the NPU SPI
   (151); confirm the real MMU fault line on ROCK 3B and split it if needed.
3. **Vision model** — ship / point to a `mobilenetv2-12_rk3568.rknn`
   (target_platform must be `rk3568`; an rk3576 `.rknn` will be rejected).
4. **NPU clock rate** — 800 MHz is the assigned rate; confirm the OPP/voltage
   the ROCK 3B PMIC actually supports.
